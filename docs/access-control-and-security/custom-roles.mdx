# Managing Custom Roles

:::note
This feature is only available to Admins.
:::

In Orkes Conductor, custom roles let you define fine-grained access control to your Conductor resources. They allow you to create roles that include only the permissions your users, groups, or applications need, without relying solely on the broader system roles. A user can have multiple roles assigned, and their effective access is the union of all permissions across all assigned roles.

:::tip 5-minute path
Go to Access Control > Roles, select + Create Role, name it in uppercase with underscores, select the resource and access-control permissions it needs, then select Create Role.
:::

## Configuring custom roles

Configure custom roles to control what users, groups, or applications can do and what resources they can access.

:::note
Every custom role automatically includes Event Handlers – Create and User Forms – Create, and these permissions cannot be removed.
:::

**To configure a custom role:**

1. Go to **Access Control** > **Roles** from the left navigation menu on your Conductor cluster.
2. Select **+ Create Role**.
3. In **Role name**, enter a unique name. Must be uppercase with underscores between words. You cannot rename a role after creating it.
4. Select the permissions to include in the role. Permissions are organized into two sections:
    - **Resource Permissions**: Controls what actions the role can perform on each resource type.
    - **Access Control**: Controls visibility into applications, groups, roles, and users.
5. Select **Create Role**.

### Resource permissions

The following table describes what each action does per resource type:

| Resource | Create | Read | Update | Delete | Execute |
| --- | --- | --- | --- | --- | --- |
| Agents | Creates a new agent | List and view agents | Edit agents | Delete an agent | Start or run an agent |
| AI Prompts | Creates a new prompt | List and view prompts | Edit prompt, manage tags | Delete a prompt or version | N/A |
| API Gateway Services | Create a new gateway service | List and view gateway services | Edit gateway service, manage tags | Delete services and all its routes | N/A |
| Auth Configs | Create an auth config for gateway services | List and view auth configs | Edit auth config | Delete auth config | N/A |
| Environment Variables | Create a new environment variable | List and view environment variables | Update an environment variable | Delete an environment variable | N/A |
| Event Handlers | Create a new event handler | List and view event handlers | Edit event handlers, manage tags | Delete an event handler | N/A |
| Integrations | Add a new integration provider | List and view integration providers | Edit an integration provider, manage tags | Delete an integration provider | Use the integration in workflow executions |
| Schemas | Create a new schema | List and view schemas | Update schema (creates a new version) | Delete a schema or a specific version | N/A |
| Secrets | Create a new secret | List and view secrets | Update/rotate a secret value | Delete a secret | N/A |
| Task Definitions | Create a new task definition | List and view task definitions | Edit task definitions, manage tags | Delete a task definition | Required to execute workflows containing this task definition |
| User Forms | Create a new user forms | List and view user forms | Edit user forms, manage tags | Delete a user form or a specific version | N/A |
| Workflow Definitions | Create a new workflow definition | List and view workflow definitions | Edit workflow definition, manage tags | Delete a workflow definition or version | Start or run a workflow |
| Workflow Schedules | Create a new schedule | List and view schedules | Edit schedule, pause/resume schedule, manage tags | Delete a schedule | N/A |

:::note
Creating a workflow schedule does not grant access to the workflow definition it targets. You must have Execute on the workflow definition separately.
:::

### Access control permissions

The following table describes what each action does per resource type:

| Resource | Create | Read | Update | Delete |
| --- | --- | --- | --- | --- |
| Applications | Create a new application | List and view applications | Edit application, create access keys, assign roles | Delete application and revoke all its permission |
| Groups | N/A | List and view groups and their members | N/A | N/A |
| Roles | N/A | List and view roles and their permissions | N/A | N/A |
| Users | N/A | List and view users | N/A | N/A |

## Permission behavior

Keep the following in mind when configuring permissions:

- **Create**: When a user creates a resource, they automatically get full access to that resource (read, update, delete, and execute where applicable). They cannot see resources created by others unless Read is also granted.
- **Read** is required for org-wide listing. Without Read, a user can only see resources they created themselves.
- **Update**, **Delete**, and **Execute** each automatically include Read on the same resource type, giving org-wide read on all instances of that resource.
- **Update** does not imply Delete, and vice versa. Each must be granted explicitly.

## Using custom roles

You can assign custom roles to:

- **Users** – to restrict access for individual accounts.
- **Groups** – to apply consistent permissions across multiple users.
- **Applications** – to scope what API keys can access.

<details>
<summary>Assigning a custom role to a user</summary>

**To configure a custom role to a user:**

1. Go to **Access Control** > **Users** from the left navigation menu on your Conductor cluster.
2. Create a new user or select an existing one.
3. In **Roles**, select the previously created custom role.
4. Select **Save**.

</details>

<details>
<summary>Assigning a custom role to a group</summary>

**To configure a custom role for a group:**

1. Go to **Access Control** > **Groups** from the left navigation menu on your Conductor cluster.
2. Create a new group or select an existing one.
3. In **Assign roles**, select the previously created custom role.

</details>

<details>
<summary>Assigning a custom role to an application</summary>

**To configure a custom role for an application:**

1. Go to **Access Control** > **Applications** from the left navigation menu on your Conductor cluster.
2. Create a new application or select an existing one.
3. In **Custom roles**, select the previously created custom role.

</details>

While editing a custom role, the changes apply immediately across all users, groups, and applications assigned to the role. Application access keys automatically inherit updated permissions.

## Production notes

- Grant Read sparingly — it exposes org-wide visibility into every resource of that type, not just resources the role's holders created.
- Review Update and Delete grants separately; neither implies the other, so audit both when scoping a role.
- Treat role edits as immediately live. A change to a shared custom role affects every user, group, and application assigned to it, including running application access keys.
- Prefer custom roles over broad system roles when a team only needs access to a specific subset of resource types.
