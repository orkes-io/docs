# Idempotency in Conductor Workflows

Idempotency prevents duplicate workflow executions for the same logical request. It is essential when clients retry after timeouts, event sources deliver messages more than once, or webhooks are resent.

!!! tip "5-minute path"
    Choose a stable business key, pass it as the workflow idempotency key, choose a conflict strategy, then test two identical starts and verify the second request behaves as expected.

Use idempotency for workflows that produce side effects: payments, account creation, ticket creation, notifications, external API calls, fulfillment, or data writes.

## How idempotency works in Conductor

When a start request includes an idempotency key, Conductor checks for an existing workflow execution with the same workflow name and key.

- If no matching execution exists, Conductor starts a new workflow.
- If a matching execution exists, Conductor applies the selected idempotency strategy.

Workflow-level idempotency prevents duplicate executions. Worker code should still be idempotent because a task can be retried within one workflow execution.

### Idempotency key

An idempotency key should identify the business operation, not the retry attempt.

Good keys:

- `order-ORD-1001`
- `payment-intent-pi_123`
- `webhook-stripe_evt_123`
- `${workflow.input.eventId}` for webhook/event-triggered workflows

Avoid random UUIDs generated by the caller on each retry. A new UUID defeats deduplication.

### Idempotency strategy

| Strategy | Behavior | Use when |
| -------- | -------- | -------- |
| `FAIL` | Starts only if no matching execution exists. Duplicate requests fail. | You need strict once-only behavior, such as payment capture. |
| `RETURN_EXISTING` | Returns the existing workflow ID and does not start a duplicate. | Clients retry on timeout and can reuse the existing execution. |
| `FAIL_ON_RUNNING` | Fails only when a matching execution is `RUNNING` or `PAUSED`; completed executions can run again. | You want no overlap but allow later reruns, such as scheduled jobs. |

## Configuring idempotency

### Via the Start Workflow API

Pass the idempotency key and strategy with the start request.

```bash
curl -X POST "https://<YOUR-SERVER-URL>/api/workflow/order_processing" \
  -H "Content-Type: application/json" \
  -H "X-Authorization: <TOKEN>" \
  -H "X-Idempotency-key: order-ORD-1001" \
  -H "X-on-conflict: RETURN_EXISTING" \
  -d '{
    "orderId": "ORD-1001",
    "customerId": "CUST-9"
  }'
```

### Via the Java SDK

```java
StartWorkflowRequest request = new StartWorkflowRequest();
request.setName("order_processing");
request.setVersion(1);
request.setInput(Map.of("orderId", "ORD-1001"));
request.setIdempotencyKey("order-ORD-1001");
request.setIdempotencyStrategy(IdempotencyStrategy.RETURN_EXISTING);

String workflowId = workflowClient.startWorkflow(request);
```

### Via the Conductor UI

Use the UI for manual testing by providing the idempotency key and strategy in the workflow run configuration. For production clients, use the API, SDK, webhook trigger, or gateway layer so the key is generated consistently from business data.

## Idempotency for webhook-triggered workflows

For webhook workflows, derive the idempotency key from the event payload.

Example:

```json
{
  "idempotencyKey": "${workflow.input.event.id}",
  "idempotencyStrategy": "RETURN_EXISTING"
}
```

Use the source system's event ID when available. If the source does not provide a stable event ID, derive a key from immutable business fields such as order ID plus event type.

## Searching executions by idempotency key

Search by idempotency key when debugging duplicate requests. Store the key in the caller logs as well as in Conductor so support teams can trace the logical request across systems.

Useful audit fields:

- Idempotency key.
- Idempotency strategy.
- Workflow ID returned to the caller.
- Original request ID or event ID.
- Caller application or webhook source.

## Strategy decision guide

| Scenario | Recommended strategy |
| -------- | -------------------- |
| Charge exactly once | `FAIL` |
| Client retries a REST call after timeout | `RETURN_EXISTING` |
| At-least-once event or webhook delivery | `RETURN_EXISTING` |
| Scheduled job should not overlap | `FAIL_ON_RUNNING` |
| Manual reruns should be allowed after completion | `FAIL_ON_RUNNING` |

## Best practices

- Use a stable business identifier as the key.
- Keep the key scoped to the operation, not the transport retry.
- Make workers idempotent even when workflow starts are idempotent.
- Test duplicate starts in CI or staging.
- Combine idempotency with workflow search and correlation IDs for supportability.
- Use terminal worker failures for invalid duplicate requests that should never retry.
